FIX 1 — STATUS DETECTION (any language, any format)
Status may be expressed as a label, a word, a percentage, a sentence, or implied context. Reason from all available signals:
- Explicit completion in any language (Terminado, done, finished, 100%, cerrado, concluido) → completed
- Explicit active progress in any language (En Proceso, in progress, ongoing, active) → ontack
- Explicit blockage (Bloqueado, En Espera, waiting on, pending approval, can't proceed until) → blocker
- Explicit risk (En Riesgo, at risk, behind, overdue, atrasado, retrasado) → atrisk
- Not yet started (No empezado, not started, 0%) → ontack with low pct
- Implied difficulty or stalling from narrative language → atrisk or blocker based on context
Do not default to ontack when the narrative gives you signals pointing elsewhere.

FIX 2 — PERCENTAGE-DRIVEN CATEGORIZATION
pctComplete overrides the status label:
- pctComplete === 100 → ALWAYS "completed", regardless of label or language
- pctComplete >= 80 → likely "completed" or nearly so; confirm with observations text
- pctComplete <= 15 AND task has a due date within 14 days → escalate to "atrisk"
- pctComplete === 0 AND task appears in 3+ consecutive reports → flag as potential stall
Never let a label of "In Progress" keep a task at "on_track" if pctComplete is 100.

FIX 3 — DUE DATE EXTRACTION AND SORT PRIORITY
Extract every date associated with a task: start date, target/delivery date, last-modified date. Use the delivery or target date as the due date. Apply Rule 3 (OVERDUE flagging) and Rule 4 (urgency sort) from LARGE LIST HANDLING above.

FIX 4 — START DATE CONTEXT (STALL DETECTION)
If a task has a startDate older than 60 days ago AND pctComplete < 30, it is a potential stall. Flag it as "atrisk" with text: "Possible stall — started [X] days ago, only [N]% complete". Do NOT silently leave it as "on_track". This applies even if the status label says "In Progress".

FIX 5 — STRICT CATEGORY SEPARATION AND DEDUPLICATION
Assign each task to exactly one highlight type using this priority order. The first matching rule wins:
  1. pctComplete === 100, or status is Terminado/Completado/Finalizado → type: "completed" — COMPLETED ALWAYS WINS, regardless of any due date
  2. Actively blocked (externally blocked, waiting on authorization/approval, Bloqueado, En Espera) → type: "blocker"
  3. Is overdue (due date < summaryDate and not complete) → type: "atrisk", prepend "OVERDUE · " — NEVER "ontack"
  4. pctComplete <= 15 with a near deadline (≤14 days), or stalled (per Fix 4) → type: "atrisk"
  5. Exceptional result, big metric, standout win → type: "standout"
  6. Plans for tomorrow, next-day objectives → type: "tomorrowfocus"
  7. Everything remaining that is active → type: "ontack"
A task MUST appear in exactly one category. Never emit the same task in two highlights. The category sort order (standout → completed → atrisk → ontack → blocker → critical → tomorrowfocus) is fixed and is applied AFTER categorization — this pipeline only determines the type, not the display order.

FIX 6 — APPLY TO ALL FORMATS
These reasoning rules apply whether the source data came from a PDF project table, a plain text activity log, an Excel sheet, or a narrative summary. The format of the source document does not change how tasks are interpreted and categorized.
